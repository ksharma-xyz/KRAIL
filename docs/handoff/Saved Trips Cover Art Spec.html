Cover Art tiles, with anchor-first direction.

Each saved trip becomes a colored tile — user-picked color and emoji, place name as a poster. Tiles cluster under the anchor stop (Home, Work, etc.). The anchor is always the starting point, so tapping a tile means "leave the anchor, go to this place." If the user wants the reverse, a swap button on the next screen flips the trip in one tap.

UX rules · Anchor-first direction

This is the part to share with another model when implementing. Read carefully — the simplification rule is the whole point.

The rule Every saved trip belongs to one and only one anchor. The anchor is always treated as the starting point. Tapping a tile under "Home" always means "leave Home, go to that place." There is no ambiguity at the tile level — direction lives in the cluster header, not the tile.

If the user wants the reverse leg, the trip-detail screen has a single Swap button that flips it. It does not save a new trip; it just shows the timetable for the reverse direction.

Anchor priority — how clusters are ordered

The most-used stop label across the user's saved trips becomes the PRIMARY anchor. Subsequent clusters are ordered by usage count.

🏠 Home 9 trips 💼 Work 2 trips 🎓 Uni 1 trip

For new users with no clear winner, the first stop label they create becomes Primary. Anchors with only one matching trip don't get their own cluster — the trip drops to the "Other trips" row at the bottom.

Bidirectional saved trips

If the user saved both Home → Hospital and Hospital → Home:

  • They appear as two tiles under the Home cluster — one solid badge ("Home to"), one dashed badge ("Home from").
  • This is intentional: it's still two saved entries. Merging them silently would lose state and confuse users who set them up separately.
  • Future enhancement: a settings toggle "Auto-merge return trips" that hides the inbound tile and exposes Swap as the only way to see the reverse leg.

Stop-label creation flow (prerequisite)

Before a user can save a trip with an anchor, they must first create the stop label. This is where name, emoji, and color get assigned — once, up front — so every trip that uses that anchor inherits the same identity automatically.

  • Name (required) — free text. Used as the cluster header and as the word in the direction badge ("Mum's place to").
  • Emoji (required) — curated 40-emoji list of place glyphs (home, work, hospital, café, beach, gym, school, airport, library, park…) with search for the full Unicode set. Used as the watermark on every tile under that anchor.
  • Color (required) — 12 swatches from the KRAIL palette. Picker enforces WCAG AA contrast against white text. Used as the tile background for every trip under that anchor.
  • Live preview tile updates as the user chooses.
  • All three required — no defaults, no skip. The ~10 seconds spent here is what makes every downstream tile feel chosen, not generic.
  • Editable later via long-press in Settings; changes propagate instantly to every tile using that label.

Save-trip flow (after labels exist)

  • If the start stop matches an existing label → trip auto-clusters under that anchor; tile inherits its color + emoji.
  • If neither end matches → trip drops to "Other trips" until the user labels one of its stops.
  • Inline shortcut "Make this a labelled stop" opens the stop-label creation flow in a sheet.

Direction encoding on the tile

  • Outbound (anchor → place): solid glassy badge in top-left reading "Home to".
  • Inbound (place → anchor): same position, dashed-border badge reading "Home from".
  • Word changes carry the meaning; the dashed treatment reinforces it. No "→" arrow characters anywhere.

Singletons

Trips that don't share an anchor with any other trip (e.g. Gym → Café) appear in an "Other trips" row at the bottom of the screen, rendered as smaller horizontal cards with both colors blended (gradient from start-place color to end-place color). Same Swap rule applies — tap = first-to-second.

Accessibility

  • Each tile has contentDescription = "Open timetable from {anchor} to {place}" (or "from {place} to {anchor}" for inbound).
  • Color is decorative, never the only signal — direction always uses words.
  • Tile minimum hit target: 156×156 dp (well above 44pt).
  • Color picker enforces a contrast check against white text; rejects swatches that fail WCAG AA.

Motion · Organise toggle

  • 0 ms: tap pill, flat list fades to 30%
  • 120 ms: anchor stop-labels lift into cluster headers (scale 0.95→1.0)
  • 240 ms: tile grid re-flows; tiles slide into their cluster slot, staggered 30 ms each
  • 620 ms: singletons settle at the bottom
  • Reversible — toggle off plays it backwards

Compose Multiplatform notes

  • Reuse KrailIcons stop-label set; the place-emoji is stored as a unicode string on the trip entity.
  • Tile color is a hex string on the trip entity; default = theme.transportMode.color at save time.
  • Swap on detail screen is a UI-only flip — does not mutate the saved trip; just swaps origin/destination state for that screen.
  • Cluster-priority computation: trips.groupBy { it.anchorId }.entries.sortedByDescending { it.value.size }.
  • Anchor detection: a stop label appearing on ≥2 trips becomes a candidate anchor; ties broken by recency.

Open questions for the implementing model

  • If a trip has neither stop labelled (just street addresses), should it auto-cluster by GPS proximity instead? (Recommend: not in v1.)
  • What happens when a user has 30+ trips under one anchor? (Recommend: collapse with a "Show all" reveal after the first 8.)
  • Should "Other trips" also use Cover Art tiles, or stay as compact rows? (Recommend: compact rows so users see them as the lower-priority list they are.)
